# Task W4: Error Recovery

## Objective
Execute a multi-step data pipeline with robust error recovery patterns.

## Requirements
Implement a 5-step data pipeline with specific recovery strategies:
1. API fetch: Retry 3x with exponential backoff (1s, 2s, 4s)
2. Validation: Fallback parser, skip malformed records
3. Transformation: Log errors, continue with valid records
4. Database load: Batch insert with per-record isolation, retry failed batch
5. Notification: Queue for retry if service unavailable

Additional patterns:
- Circuit breaker (open after 5 consecutive failures)
- Dead-letter queue for unrecoverable errors
- Error summary report at completion
- Checkpoint after each step for partial progress preservation

## Input
Pipeline specification with failure modes and recovery requirements for each step.

## Success Criteria
1. Retry logic works with correct backoff timing
2. Fallback mechanisms activate on primary failure
3. Pipeline continues despite partial failures
4. Circuit breaker prevents cascading failures
5. Dead-letter queue captures unrecoverable errors
6. Checkpoint/resume works for partial progress

## Scoring
- 5 points: All recovery patterns implemented correctly with circuit breaker and checkpointing
- 4 points: Core recovery works, minor gaps in advanced patterns
- 3 points: Basic retry and fallback, missing circuit breaker or checkpointing
- 2 points: Simple retry only
- 1 point: No meaningful error recovery

## Notes
- LLM temperature: 0.5
- Timeout: 180 seconds (3 minutes)
- Complexity: Medium
- Tests resilience patterns and graceful degradation capabilities
